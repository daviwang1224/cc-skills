## 访谈原则

使用 AskUserQuestion 工具进行 2-3 轮深度访谈，每轮 1-3 个问题。

### 核心原则

1. **问题要深入**：不问显而易见的问题
2. **问题要具体**：避免过于宽泛的提问
3. **问题要关键**：聚焦核心决策点和不确定性
4. **轮次要适度**：2-3 轮即可，避免过度提问

### 提问技巧

- **开放式问题**：了解背景和上下文
- **封闭式问题**：确认具体细节和选择
- **追问式问题**：深入挖掘真实需求
- **假设式问题**：探索边界条件和特殊情况

## 访谈维度

根据场景选择合适的提问维度：

### 1. 技术实现维度

**关注点**：
- 现有技术栈和约束
- 性能和规模要求
- 技术债务和历史包袱

**典型问题**：
- 当前使用的技术栈是什么？
- 有哪些技术约束或限制？
- 预期的性能指标是什么？
- 是否有遗留系统需要兼容？

### 2. 用户体验维度

**关注点**：
- 目标用户和使用场景
- 交互方式和体验预期
- 可访问性和易用性

**典型问题**：
- 目标用户是谁？他们的技术水平如何？
- 主要使用场景是什么？
- 对用户体验有什么特殊要求？
- 是否需要考虑无障碍访问？

### 3. 业务目标维度

**关注点**：
- 核心价值和优先级
- 成功指标和验收标准
- 时间和资源约束

**典型问题**：
- 这个功能要解决什么业务问题？
- 如何衡量成功？有哪些关键指标？
- 优先级如何？是否有时间限制？
- 可用的资源（人力、预算）是多少？

### 4. 风险权衡维度

**关注点**：
- 可接受的风险和代价
- 关键的非功能性需求
- 未来扩展性考虑

**典型问题**：
- 最不能接受的风险是什么？
- 对可用性、安全性、性能的要求如何排序？
- 未来可能的扩展方向是什么？
- 可以接受的技术债务程度是多少？

## 访谈示例

### 示例1：认证系统设计

**场景**：用户要求设计认证系统

#### 第1轮访谈（理解上下文）

```
问题1：这个认证系统是给什么类型的应用使用？
- 选项A：Web应用
- 选项B：移动应用
- 选项C：API服务
- 选项D：多端应用（Web + 移动 + API）

问题2：预期的用户规模和并发量级是多少？
- 选项A：小规模（<1000用户，<100并发）
- 选项B：中等规模（1000-10000用户，100-1000并发）
- 选项C：大规模（>10000用户，>1000并发）

问题3：是否有特殊的安全合规要求？
- 选项A：无特殊要求
- 选项B：需要符合行业标准（如GDPR、等保）
- 选项C：有严格的安全审计要求
```

#### 第2轮访谈（深入细节）

根据第1轮回答，假设用户选择了"多端应用 + 中等规模 + 行业标准"：

```
问题1：是否需要支持第三方登录（OAuth）？
- 选项A：不需要，只用账号密码
- 选项B：需要，支持常见平台（微信、支付宝等）
- 选项C：需要，支持企业SSO（如LDAP、SAML）

问题2：对于会话管理，更倾向于哪种方式？
- 选项A：有状态（Session + Cookie）
- 选项B：无状态（JWT）
- 选项C：混合方式（根据场景选择）

问题3：是否需要支持多设备同时登录？
- 选项A：不需要，一个账号只能一个设备登录
- 选项B：需要，但有设备数量限制
- 选项C：需要，不限制设备数量
```

#### 第3轮访谈（可选，根据需要）

如果前两轮回答中有不确定的地方，可以继续追问：

```
问题1：对于JWT的过期时间，有什么偏好？
- 选项A：短期（15分钟-1小时）+ Refresh Token
- 选项B：中期（1-24小时）
- 选项C：长期（>24小时）

问题2：是否需要实现"记住我"功能？
- 选项A：需要，用户可以选择长期登录
- 选项B：不需要，每次都要重新登录
```

### 示例2：性能优化

**场景**：用户反馈API响应慢

#### 第1轮访谈（定位问题）

```
问题1：慢到什么程度？
- 选项A：偶尔慢（<10%请求>1s）
- 选项B：经常慢（>50%请求>1s）
- 选项C：一直慢（平均响应>2s）

问题2：是所有接口都慢，还是特定接口慢？
- 选项A：所有接口都慢
- 选项B：只有某些特定接口慢
- 选项C：不确定，没有详细监控

问题3：你已经做过哪些排查？
- 选项A：还没开始排查
- 选项B：看过日志，但不确定问题
- 选项C：定位到某个环节慢，但不知道怎么优化
```

#### 第2轮访谈（深入分析）

根据第1轮回答，假设用户选择了"特定接口慢 + 定位到数据库查询慢"：

```
问题1：这个慢的接口的查询特点是什么？
- 选项A：查询数据量大（返回大量记录）
- 选项B：查询条件复杂（多表JOIN）
- 选项C：查询频率高（高并发）

问题2：数据库是否有索引？
- 选项A：有索引，但不确定是否合理
- 选项B：没有索引
- 选项C：不清楚

问题3：是否可以接受缓存方案？
- 选项A：可以，数据实时性要求不高
- 选项B：不可以，必须实时数据
- 选项C：部分数据可以缓存
```

### 示例3：功能设计

**场景**：用户要求添加搜索功能

#### 第1轮访谈（理解需求）

```
问题1：搜索的主要对象是什么？
- 选项A：文本内容（文章、评论等）
- 选项B：结构化数据（商品、用户等）
- 选项C：混合内容（文本 + 结构化数据）

问题2：预期的数据规模是多少？
- 选项A：小规模（<10万条记录）
- 选项B：中等规模（10万-100万条记录）
- 选项C：大规模（>100万条记录）

问题3：对搜索功能的期望是什么？
- 选项A：基础搜索（精确匹配或简单模糊匹配）
- 选项B：智能搜索（支持分词、相关性排序）
- 选项C：高级搜索（支持过滤、聚合、推荐）
```

#### 第2轮访谈（技术选型）

根据第1轮回答，假设用户选择了"文本内容 + 中等规模 + 智能搜索"：

```
问题1：是否可以接受搜索结果的延迟？
- 选项A：可以接受几秒延迟（异步索引）
- 选项B：需要准实时（<1秒延迟）
- 选项C：必须实时（即时索引）

问题2：对搜索准确性的要求如何？
- 选项A：高准确性（宁可漏掉也不要错误结果）
- 选项B：高召回率（宁可多返回也不要漏掉）
- 选项C：平衡准确性和召回率

问题3：是否需要支持中文分词和拼音搜索？
- 选项A：需要中文分词
- 选项B：需要拼音搜索
- 选项C：两者都需要
- 选项D：都不需要
```

## 访谈最佳实践

### ✓ 应该做的

1. **循序渐进**：从宏观到微观，从简单到复杂
2. **聚焦关键**：只问影响决策的关键问题
3. **提供选项**：给出具体选项帮助用户思考
4. **灵活调整**：根据回答动态调整后续问题
5. **适时停止**：获得足够信息后及时结束访谈

### ✗ 不应该做的

1. **过度提问**：不要超过3轮访谈
2. **问题过宽**：避免"你想要什么功能"这类问题
3. **问题过细**：不要在访谈阶段问实现细节
4. **重复提问**：不要问用户已经说过的内容
5. **假设答案**：不要在问题中暗示期望的答案

## 访谈后的处理

完成访谈后，应该：

1. **总结理解**：用简洁的语言复述用户需求
2. **识别意图**：分析用户的真实意图和隐藏需求
3. **选择思维模型**：根据场景选择合适的思维方式
4. **开始深度分析**：运用思维模型进行系统分析
